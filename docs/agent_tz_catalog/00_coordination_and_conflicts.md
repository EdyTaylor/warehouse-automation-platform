# Координация работ по каталогу цен

## Цель
Разделить доработку каталога на независимые блоки для параллельной работы нескольких агентов с минимальными merge-конфликтами.

## Текущее состояние ветки (перед стартом)
- Уже изменены: `stock_operations.php`, `sync_monitor.php`, `b24_sales.php`.
- Эти файлы считаются "горячими" и не должны редактироваться несколькими агентами одновременно.

## Разделение задач по агентам
- Агент A: UI/UX каталога и inline-редактирование.
- Агент B: Валидация/правила цен и превью итоговой цены.
- Агент C: Синхронизация каталога с B24 (очередь, статусы, retry).
- Агент D: История изменений цен и аудит.

## Владелец файлов (чтобы не конфликтовать)
- Агент A владеет:
  - `products.php`
  - `assets/style.css` (только секции для каталога)
- Агент B владеет:
  - `functions/pricing.php`
  - `products.php` (только блоки server-side validation, строго по маркерам)
  - `warehouse_orders.php` (только read-only проверка; без изменений в MVP)
- Агент C владеет:
  - `products.php` (только блоки sync action/endpoints, строго по маркерам)
  - `api/bitrix/send.php` (только если нужен безопасный retry helper)
  - `functions/app_settings.php` (только новые settings ключи для sync)
- Агент D владеет:
  - `migrations/008_product_price_history.sql` (новая миграция)
  - `products.php` (только вставка логирования после save)

## Основные точки риска конфликтов
- `products.php` — общий файл для A/B/C/D.
  - Решение: каждый агент меняет только свой помеченный блок:
    - `// [CATALOG_UI_BLOCK_START/END]`
    - `// [CATALOG_VALIDATION_BLOCK_START/END]`
    - `// [CATALOG_B24_SYNC_BLOCK_START/END]`
    - `// [CATALOG_AUDIT_BLOCK_START/END]`
- `b24_sales.php` — уже модифицирован под совместимость PHP.
  - Решение: не трогать в рамках каталога.
- `stock_operations.php`/`sync_monitor.php` — уже модифицированы под валюту.
  - Решение: не трогать в рамках каталога.

## Порядок интеграции PR
1. Агент B (валидация) — задает правила данных.
2. Агент D (аудит) — фиксирует изменения.
3. Агент C (B24 sync) — навешивает надежный sync/retry.
4. Агент A (UI) — финально подключает сценарии управления на интерфейсе.

## Общие требования ко всем агентам
- Совместимость с текущим PHP (без `??` и современных конструкций, если есть риск).
- Не ломать существующий fallback из `resolveTierPrice()`.
- Любое изменение цен должно оставлять локальное сохранение рабочим даже при ошибке B24.
- Любые массовые операции должны иметь dry-run превью перед применением.
