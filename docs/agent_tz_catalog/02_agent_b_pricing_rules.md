# ТЗ для Агента B: правила цен, валидация, превью итоговой цены

## Цель
Убрать ценовые ошибки до синка и до работы кладовщика: обеспечить прозрачное применение tier/fallback.

## Объем работ
1. Усилить `functions/pricing.php`:
   - сохранить текущую логику `resolveTierPrice()`;
   - добавить helper для объяснения "какая цена применится и почему".
2. В `products.php` добавить server-side валидацию:
   - запрет отрицательных значений;
   - предупреждение о пустых tier-полях;
   - предупреждение о нетипичных скачках между tier-ценами;
   - корректная нормализация дробных значений.
3. Добавить в UI превью:
   - "Цена при 3/7/12/25 рулонах";
   - отображение tier-source (`price_1_4`, fallback и т.д.).
4. Добавить "быстрые автозаполнения":
   - если `price_5_9/10_19/20+` пустые, предложить заполнить по цепочке от `price_1_4`.

## Ограничения
- Не делать B24 sync-логику (агент C).
- Не делать историю изменений (агент D).
- Не выполнять глубокий редизайн UI beyond валидационных подсказок (агент A).

## Технические требования
- Основной файл: `functions/pricing.php`.
- В `products.php` менять только блоки валидации и превью.
- PHP-совместимость с текущим окружением (без рискованных синтаксических новинок).

## Acceptance Criteria
- Невалидная цена не попадает в сохранение без понятной ошибки.
- Пользователь заранее видит итоговую цену для ключевых tier-диапазонов.
- Fallback в превью соответствует фактической логике, используемой в заказах.
